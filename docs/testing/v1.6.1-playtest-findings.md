## F1 — Session toolbar button didn't render on Foundry v13 build 350 · 🟢 resolved (external)

**Environment:** The Forge, Foundry VTT v13 build 350.

**Observed:** The ▶ Session button (`sfSession`) did not appear in the Token
Controls toolbar, even though the other Companion buttons (🛡 Companion Settings,
Progress Tracks, Entities, Clocks) did. Updating Foundry from build **350 → 351**
made it appear.

**Module side:** No defect found. The registration matches the canonical two-hook
v13 pattern (`getSceneControlButtons` registers metadata, `renderSceneControls`
binds the click) in `src/index.js`, and `sfSession` has **no `visible` gate**, so
it should show for all users. The v1.6.1 commits only touched comment text in
`src/index.js`, never the toolbar code.

**For later:** if it recurs on a specific v13 build, investigate whether the
first-registered tool or a tool lacking an explicit `order` is dropped by that
build's scene-controls renderer.

## F3 — Sector creation: settlement Actors created but sheets/icons not populated · 🔴 open

**Environment:** fresh world, `!sector new` → sector "Onyx Verge".

**Observed:**
- Four settlement Actors were created — **Argosy, Nyx, Sirius, Welkin** — under
  Actors → **Sectors → Unsorted**, using the generic foundry-ironsworn location
  sheet.
- Those Actor sheets are **empty**: Region, Location type, Type of settlement, and
  the roll tables (Population, First look, Initial contact, Authority, Settlement
  projects, Settlement trouble) are all unfilled. **No portrait icon** is set
  (generic hooded silhouette).
- Meanwhile the journal **"Onyx Verge — Sector Record"** captures everything well:
  Region (Terminus — the settled core), Trouble (settlements/factions on the brink
  of war), a narrative blurb, and a Settlements list with full data —
  Nyx (Deep Space, Pop: Hundreds, Authority: Unyielding),
  Welkin (Orbital, Pop: Tens of thousands, Authority: None/lawless),
  Argosy (Orbital, Pop: Thousands, Authority: Corrupt),
  Sirius (Planetside, Pop: Thousands, Authority: Corrupt) — plus "4 passages charted".
- Reporter notes the flow "only partially uses the Starforged UI / continues to run
  duplicates" — settlement data appears to live in two representations (the
  unsorted settlement Actors **and** the Sector Record journal under
  Sectors → Onyx Verge) without the Actor sheets being filled.

**Desired outcome (reporter):** the settlement **Actor cards should be filled out**
(sheet fields populated) and **icons set**, *in addition to* the journal snippet.

**Likely-related (for diagnosis, not yet investigated):**
- `docs/entities/entity-actor-migration-scope.md` (📋 planned) — makes
  settlement/location entities **native Actors as the source of truth**, collapses
  the settlement-data duplication across storage layers, and adds hierarchical
  Actor folders. This finding looks like the gap that scope is meant to close.
- `docs/sectors/sector-creator-enhanced-scope.md` and `src/sectors/sectorGenerator.js`,
  `src/sectors/sectorOverview.js` — how settlement Actors vs the Sector Record are
  produced.
- `docs/foundations/system-asset-integration-scope.md` + `src/system/ironswornAssets.js`
  — the icon resolvers (`statIcon` / `assetIcon` / location icons). The scope-index
  "Visual UI polish" next-step notes these resolvers shipped but were never wired
  into the panels/cards.
- `src/entities/settlement.js`, `src/entities/registry.js`, `src/entities/folder.js`
  — settlement record CRUD, host-document dispatch, and the "Unsorted" vs per-sector
  folder placement.

## Theme T1 — an explicit "entity created & assigned" lifecycle gate · 🟡 candidate direction

Across F3 / F4 / F6 / F7 a pattern emerges: the module **generates names, flavor,
art, and journal/Notes stubs eagerly at creation time**, before the entity is
named, assigned, and carrying its real details (assets, type, sheet fields). The
result is placeholder-name references ("Ship") and invented details (fighters,
oversized engines, off-sector settlements) that don't match the finished entity.

**Reporter's framing:** *"I'm starting to feel like we need to have a process for
declaring the ship 'created' and assigned."*

**Candidate direction (not yet actioned):** an explicit "entity finalized" step —
analogous to the Session **Begin** gate — so generation (flavor / art / Notes /
stub) fires **after** naming + assignment and is **grounded in the actual sheet
data**. Relates to `entity-actor-migration-scope.md` (native Actors as the source
of truth) and the generate-once/locked portrait lifecycle in `../decisions.md`.

**Confirmed behavior (not a bug):** a starship carrying the **Command Vehicle**
asset shows an **INTEGRITY** meter (5 → 0) on its sheet; modules (e.g. Research
Lab) appear as their own asset cards.
