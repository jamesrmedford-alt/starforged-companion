# v1.7.10 Playtest Findings

Five findings from the v1.7.10 playtest (2026-06-12, sector "Ferrous Chasm",
PC Kayla Vayan, ship Kobayashi 8). All five were diagnosed to root cause the
same evening and fixed together on `claude/admiring-carson-qlzr7h` (this PR).

Status key: 🟢 fixed/resolved · ✅ positive.

---

## F1 — Sector NPC card opens with the classic Ironsworn sheet · 🟢 fixed

**Reporter:** *"the NPC created for a sector is using the Ironsworn character
card, rather than the starforged one, for some reason"* (screenshot: "Patch
Hawking" sheet showing the Bonds track, Conditions / Banes / Burdens).

**Root cause:** foundry-ironsworn registers `IronswornCharacterSheetV2` with
`makeDefault: true` for **all** `character`-type actors; the Starforged sheet
is a non-default alternative (vendor `src/index.ts`). The system's own
"Create Starforged character" dialog works around this by pinning a per-actor
override — `flags.core.sheetClass = 'ironsworn.StarforgedCharacterSheet'` —
on every PC it creates. Our NPC cards are `character` actors too (FOLDER-002)
but `createConnection` / `migrateJournalConnectionsToActors` created them
bare, so they fell to the system default.

**Fix:** both `Actor.create` sites pin `core.sheetClass` (constant
`STARFORGED_CHARACTER_SHEET` in `src/entities/connection.js`); a ready-time
repair (`backfillNpcCardSheets`, `src/entities/migrator.js`) pins existing
NPC cards that have no explicit sheet override. Cards a GM deliberately
re-sheeted are left alone. PCs are never touched (entityType flag gate).

## F2 — Entities panel shows the stale pre-rename ship name · 🟢 fixed

**Reporter:** *"the entities panel is still listing the ship as 'Ship', even
though I re-named it to Kobayashi 8. Clicking History populates notes on it
that reference Kobayashi 8, but don't update the name."* Follow-up: *"It
appeared to update after hitting finalize."*

**Root cause:** entity flag records denormalise `name` at registration time
(`registerStarshipActorLight` snapshots `actor.name`); renaming the Actor in
the sidebar touched only `actor.name` — nothing mirrored it into the record.
The panel preferred the record (`data.name ?? document.name`), while
Envision/History read `actor.name` directly — hence the split the reporter
saw. Finalize "fixed" it because `seedStarshipActor` re-snapshots
`name: actor.name` (the design comment there already said actor.name is
authoritative).

**Fix:** live `updateActor` hook (`syncEntityRecordNameOnUpdate`,
`src/index.js`) mirrors renames into the flag record for all five
actor-hosted entity types; ready-time repair (`syncEntityRecordNames`,
`src/entities/migrator.js`) reconciles drift that predates the hook; the
Entities panel now prefers the host document name (`document.name ??
data.name`).

## F3 — All stars in a sector are identical · 🟢 fixed

**Reporter:** *"a very modest nit: all of the stars that populate a sector
appear to be identical"* (screenshot: three identical Corrupted Star pins,
one labelled "Corrupted star radiating with unnatural light").

**Root cause:** not an icon bug — `iconForStellarObject` maps all twelve
stellar results to per-type system tokens correctly (assets verified on
disk). The v1.7.1 F7 fix (commit `948fedc`) rolled `SPACE.STELLAR_OBJECT`
**once per sector** and stamped the same result on every settlement, so the
pins were genuinely identical (label and art). The original scope said "for
each settlement, optionally roll on Stellar Object"
(`docs/sectors/sector-creator-scope.md`).

**Fix:** `generateSector` rolls per settlement (each settlement is its own
star system); the unit test that pinned the shared roll now pins per-
settlement independence.

## F4 — NPC portrait/intro missing from the Notes tab · 🟢 fixed (consequence of F1)

**Reporter:** *"maybe related to the ironsworn/starforged card, but the NPC
art doesn't show up in the notes anywhere"*

**Root cause:** exactly the reporter's hunch. The two character sheets bind
their Notes tabs to different fields — Starforged `sf-notes.vue` →
`system.notes`; classic `ironsworn-notes.vue` → `system.biography`. The
finalize seed writes oracle bullets to `system.biography` and the narrator
intro to `system.notes`, then the portrait pipeline embeds the large image
at the top of `system.notes`. On the (wrong) classic sheet the Notes tab
showed the biography field, so the portrait + intro were present in the data
but rendered nowhere.

**Fix:** none needed beyond F1 — with the Starforged sheet pinned, the Notes
tab is `system.notes` and the embedded art/intro appear.

## F5 — `@scene where am I` invents a location; map token disagrees · 🟢 fixed

**Reporter:** *"still not getting a correct response to where am I. the ship
is sitting near Astra, but the narrator is putting me halfway across the map
near the vow target at Mudd."* Design direction (reporter): *"The token was
positioned on the sector. In that instance, it's position should be known.
It sounds like there is a missing hook for placing it on the sector, that
then populates the value. If it isn't on the sector, then the above
fallbacks sound great."*

**Root cause — a grounding gap plus four dormant defects.** Nothing ever
seeded the §20 ship-position record (token placement read it but never wrote
it; campaigns opening in medias res never trigger the on-movement sidecar
rule), so the narrator had no `SHIP POSITION:` line and improvised from the
premise (the vow target at Mudd). The map token near Astra was an equally
ungrounded spawn point. Investigation also found the §20 write path had
never worked live:

1. `updateShip(cv._id, …)` passed the record GUID where updateShip resolves
   an **Actor id** — every fiction-side position write threw "Ship actor
   not found" at debug level and no-op'd.
2. `inferShipPosition` / `formatShipPositionLine` read phantom
   `campaignState.settlements` record arrays that production state never
   stores (only `*Ids` exist) — name resolution always degraded to free
   text and resolved names rendered empty.
3. `resolveCurrentLocationName` (`!at`) still read journal pages —
   settlements/locations/planets are Actor-hosted post-migration, so `!at
   Astra` reported "not found".
4. The fiction→token sync's own `token.update` landed on a settlement pin
   and re-entered the preUpdateToken drag handler, which would cancel it
   and dispatch a second synthetic `set_a_course` (latent loop, unreachable
   only because no position was ever written).

**Fix (per the reporter's design):**

- **Token on a sector scene is authoritative.** New `createToken` /
  `updateToken` hooks (`src/sectors/sectorSceneHooks.js`) write the §20
  record from the token's coordinates: nearest settlement pin within the
  near radius (3× snap radius, min 3 cells) → "near <settlement>"; nothing
  in range → honest "deep space (sector)". On-pin drops keep the
  set_a_course travel-move flow; off-pin moves are position notes with no
  move mechanics.
- **Programmatic moves carry `POSITION_SYNC_OPTION`** so the hooks ignore
  the module's own token syncs (closes the latent loop).
- All §20 writers use `getCommandVehicleActorId` (new, `ship.js`);
  resolution and lookups go through the entity registry (document-first,
  in-state arrays as legacy fallback); `findSettlementNoteById` matches the
  Actor-id pin flag as well as the legacy generator id.
- **Fallbacks when no token exists:** the inciting incident's sidecar now
  REQUIRES a starting `{subject: "ship", attribute: "position"}` emission,
  and an empty position record injects an explicit *"SHIP POSITION: not yet
  established — do NOT assert or invent…"* guard line instead of silence.
- `!at <place>` works again (registry reads) and remains the manual anchor.
