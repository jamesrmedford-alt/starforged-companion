## F1 — Inciting Incident lands well · ✅ positive

**Environment:** v1.7.8, Session panel ✦ Envision Inciting Incident (first live use).

**Observed:** the card delivered exactly what the scope intended — spark

## F2 — Suggested vow should be a clickable button that creates the vow on the character · 🟡 enhancement

**Reporter:** *"the recommended vow should be a clickable button that will
create that vow in your character card."*

**Desired:** instead of (or alongside) the plain *Suggested vow:* text line on
the inciting-incident card, a button that creates that vow — statement as the
name, suggested rank — as a progress item on the player character, visible on
the character sheet's Progress tab.

**Feasibility (anchored, not yet actioned):** mostly wiring; the hard parts exist.

- The vow is already parsed structurally: `splitSuggestedVow()`
  (`src/session/incitingIncident.js:62`) yields `{ statement, rank }`.
- Creation helper already exists: `createCharacterVowItem(actor, { name, rank,
  vowId })` (`src/character/actorBridge.js:455`) — creates the
  `type: "progress"` / `subtype: "vow"` Item with rank validation (defaults
  `dangerous`) and flag-keyed idempotency.
- Button wiring must use `onChatMessageRender` (`src/system/chatHooks.js`) —
  the V13-002 dual-hook helper, never a raw `renderChatMessage` hook.
- Resolve the target PC via `getPlayerActors()` (FOLDER-002: excludes NPC
  cards, which are also `character`-type Actors).
- Store the parsed `{ statement, rank }` in the chat-message flags at create
  time so the click handler reads flags rather than re-parsing card HTML; the
  message id can serve as `vowId` for free idempotency (re-clicks no-op).
- Plus the usual: unit tests for card render + flags, Quench coverage,
  `packs/help.json` + `CHANGELOG.md` updates.

**Estimate:** small-to-medium.

## F3 — Inciting incident introduces the vow-target NPC but never instantiates them · 🔴 open

**Reporter:** *"the action is good but the followthrough is needed — the
inciting incident introduces an NPC to make a vow to, but doesn't instantiate
them in the entities ui or the actors folder or the world journal."*

**Observed:** the inciting incident invented **Vance** (estranged mentor, the
suggested vow's target) — but after the card posted:
- Entities panel: Connections shows only the pre-existing **Amelia Gray**.
- Actors sidebar: `Sectors / Igneous Spine / NPCs` contains only Amelia Gray.
- World Journal: *"No entries awaiting review"* — no pending lore, no draft.

Vance exists only in the card prose. Notably the narrator **did** ground in
established entities (real ship *Kobayashi 7*, established settlement *Lyra*)
— inventing the mentor was appropriate; the gap is purely follow-through.

**Anchor (verified in source):** `runIncitingIncident`
(`src/session/incitingIncident.js:134`) posts the card directly with **no
detection pass**. `runCombinedDetectionPass` (`src/entities/entityExtractor.js:132`)
is wired into only two paths: the move pipeline (`runPostNarrationPasses` →
`runDiscoveryDetection`, `src/narration/narrator.js:346`) and paced narration
(`runPacedDetection`, `narrator.js:1165`). The `inciting_incident` narrator
mode — like the session-vignette mode — never runs it, so nothing the opening
fiction introduces can reach the draft-Confirm flow.

**Options for triage (not decided):**
- **(a)** Run the standard combined detection pass on the inciting prose —
  Vance arrives as a pending draft; GM Confirms → NPC card (sector NPCs
  folder), Entities panel row, description persisted. Matches the module's
  GM-confirms-entities philosophy.
- **(b)** Treat the vow-target NPC as first-class: the inciting incident
  *knows* it proposed a vow to a person — auto-create (or one-click-create)
  the connection NPC card, the way `make_a_connection` / the sector wizard do.
  Could fold into **F2**'s button: swearing the suggested vow also
  instantiates its target NPC.

**Confirmed during play (later in session):** once Vance appeared in *paced*
narration (the hail exchange), the paced-path detection fired correctly —
"New Entities Detected" proposed **[Connection] Vance** and **[Ship] Unknown
contact** with Confirm/Dismiss. This pins the F3 diagnosis: detection works;
the inciting-incident path alone skips it. **Quality asymmetry:** the
paced-path draft describes Vance from the current beat only (*"ragged,
annoyed voice responding to tight-beam hail"*) — the rich inciting framing
(estranged mentor, the Sepulcher-job falling-out, failing life support) is
not in the draft. Instantiating at inciting time (option b) captures the
backstory; instantiating later captures whatever thin slice the current
scene shows. Reinforces option (b), and see **F7** for where that backstory
goes meanwhile (nowhere).

## F4 — Time-dependent suggested vows should carry a clock · 🟡 enhancement

**Reporter** (after manually creating the suggested vow on the PC): *"this
incident, autopopulated as a 'before his life support fails', feels like it
might benefit from also having a clock. Need at least a quick pass to see if
there is a time dependence there."*

**Observed:** the suggested vow statement carries explicit time pressure
(*"…before his life support fails"*), and the foundry-ironsworn vow sheet
natively supports exactly this — a **Clock** checkbox right on the progress
item (visible in the reporter's screenshot).

**Anchor (verified in vendor source at the pinned commit):**
- Progress items carry `system.hasClock` / `system.clockTicks` /
  `system.clockMax`; the sheet offers 4/6/8/10/12 segments
  (`vendor/foundry-ironsworn/src/module/vue/progress-sheet.vue:86–119`).
  A vow created by F2's button could arrive with its clock pre-attached —
  `createCharacterVowItem` would just need to accept the extra fields.
- Module-side alternative/complement: tension clocks
  (`src/clocks/clocks.js`, `createClock`) advance against Ask-the-Oracle
  odds — mechanically a good model for "life support failing" — but they
  live in the Clocks panel, not on the character sheet.

**Detection options for "is there time dependence" (not decided):**
- **(a) lead:** the narrator emits it structurally — extend the
  `inciting_incident` output contract
  (`src/narration/narratorPrompt.js:676`, currently `Suggested vow:
  <statement> (<rank>)`) with an optional `Suggested clock: <label>
  (<segments> segments)` line, emitted only when the incident has explicit
  time pressure. The same generation that wrote the deadline already knows
  it's there — zero extra API calls; parse alongside `splitSuggestedVow`.
- **(b)** heuristic regex on the vow statement ("before / until / within…")
  — cheap but brittle.
- **(c)** a Haiku classification call — overkill for one line given (a).

**Relation:** F2 / F3 / F4 are converging into one **inciting-incident
follow-through cluster**: the suggested-vow button (F2) creates the vow,
pre-attaches a clock when the vow is time-dependent (F4), and instantiates
the vow-target NPC (F3).

## F5 — Ship positioning: narrator says Lyra, token sits at Sepulcher · 🔴 open · audited

**Reporter:** *"it is clear that the ship positioning feature isn't working.
Though the ship sits near Sepulcher, the narrator believes the ship is near
Lyra. It should have either moved the Kobayashi 7 to near Lyra, or reported
the position as Sepulcher. Need to audit that feature set and see why it
isn't working."*

**Observed:** sector-scene token for *Kobayashi 7* sits beside Sepulcher's
pin; `@scene where am I in this sector` answers *"threading the outer edge
of Lyra's graveyard… Lyra settlement itself sits another eighteen thousand
klicks spinward"* — consistent with the travel fiction, contradicting the map.

**Audit result (source-verified, fact-continuity §20):** the feature is
implemented per scope and enabled by default — the mismatch falls into two
designed-in gaps, not a broken trigger.

What shipped and works:
- All four §20.4 position **writers** exist: `!at` (`src/index.js:2300`),
  chat-typed `set_a_course` non-miss via `interpretation.moveTarget`
  (`src/index.js:865–876`; `moveTarget` confirmed in
  `src/moves/interpreter.js:152,367`), narrator sidecar
  `{subject:"ship", attribute:"position"}` (`src/narration/narrator.js:1803`),
  and token drag → synthetic `set_a_course`
  (`src/sectors/sectorSceneHooks.js:99`).
- Settings all default **on** (`src/ui/settingsPanel.js:647–662`).
- The SHIP POSITION line is built into the ledger block
  (`src/narration/narratorPrompt.js:186`), confirmed-lore tier, never
  dropped — so the narrator receives whatever the record says.

**Gap 1 — no position→token sync (primary).**
`moveCommandVehicleTokenToDestination` (`sectorSceneHooks.js:222`) runs
**only** when the move originated from a token drag
(`tokenDragSetCourse` payload, `src/index.js:883`). Fiction-side movement
— chat-typed `set_a_course`, narrator sidecar, `!at` — updates the
position record but **nothing moves the token**. §20.4b's snap-back
paragraph explicitly worried about "the visual lie where the Token sits at
the destination but the persistent position says otherwise" — this is the
same lie in the reverse direction, and the scope never specified the
reverse sync. Implementation matches scope; the scope has the hole.

**Gap 2 — trigger coverage: only `set_a_course` writes position.**
The interpreter emits `moveTarget` for `set_a_course`,
`undertake_an_expedition`, and `finish_an_expedition` — but only
`set_a_course` has the §20.4 post-resolution write. If this session's
travel resolved as `face_danger` (the debris run — *"you commit to the run
past the venting reactor pod"* reads exactly like one) or
`undertake_an_expedition`, **no writer fired at all**: the record still
says Sepulcher, the SHIP POSITION block said Sepulcher, and the narrator's
"Lyra" came from scene truths / chat recency overriding the stale line.

**Gap 3 — seed-name fragility (minor).** `inferShipPosition` matches
exact name or first word (`shipPosition.js:91`); a sidecar value like
*"Lyra's orbital graveyard"* misses "Lyra" (first word "lyra's") and falls
back to freeText with **no settlement id** — narration still works, but a
future token sync would have no pin to snap to.

**Live diagnostic (10 seconds, pins which gap fired this session):** in the
browser console —
`game.actors.getName("Kobayashi 7").flags["starforged-companion"].ship.position`
— `updatedBy`/`updatedAt` provenance shows the last writer (`at_command` /
`set_a_course` / `narrator_sidecar` / `scene_token`), or a stale/Sepulcher
record meaning no writer fired (Gap 2).

**Fix directions for triage (not actioned):**
- **(a)** Position→token sync: shared `syncCommandVehicleToken(position)`
  called from both write sites (`maybeUpdateShipPositionFromName` and the
  sidecar path), reusing `moveCommandVehicleTokenToDestination` + the
  Note-pin `settlementId` flags. Free-text positions (no settlement id)
  could park the token at sector centre or leave it with a "position
  uncertain" badge — needs a decision.
- **(b)** Wire `undertake_an_expedition` / `finish_an_expedition` into the
  same §20.4 post-resolution write.
- **(c)** Possessive/suffix-tolerant name matching in `inferShipPosition`
  ("Lyra's graveyard" → "Lyra").

## F6 — @scene tone: rescue target framed adversarially · 🟡 watch

**Reporter:** *"Watching here for potential narrator drift. I'm trying to
rescue someone, but the narrator's @scene functionality makes it seem
adversarial. Could be passing, let's see if it continues."*

**Observed:** `@scene where is Vance's shuttle now relative to me` →
*"…engines dark and running cold. The shuttle's position puts it between
you and the deeper salvage, close enough that you'd cross his line of
sight if you pushed further in. He hasn't pinged you, but the way his
craft is angled suggests he's watching the approaches, not the wreckage."*
Line-of-sight / being-watched language reads like an ambush setup, not a
dying man awaiting rescue.

**Two readings (undecided — that's why it's a watch item):**
- **Defensible fiction:** the inciting incident established Vance as
  *running from something* ("he was running from it more than he was
  running to anything") and hiding wounded — "watching the approaches"
  is consistent wariness toward whatever he fled, not hostility toward
  the PC. The scene truths from the incident may be legitimately driving
  this coloring.
- **Drift:** `@scene` is a factual interrogation surface; if it keeps
  escalating tension or recasts an ally as a threat (e.g. Vance acting
  against the PC), that's the narrator importing adversarial framing the
  fiction hasn't earned.

**Escalation criteria:** log follow-ups if Vance is narrated as hostile
*toward the PC*, or if @scene answers keep adding unprompted menace.
Knobs if confirmed: `@scene` system-prompt guidance (interrogation answers
describe, they don't escalate), pacing/tone dials, scene-truth phrasing
from the inciting incident.

## F7 — NPC contradicts the inciting incident: opening fiction has no durable home · 🔴 open · root cause traced

**Reporter:** *"Still looking at narrator continuity. The NPC doesn't
acknowledge that he sent a distress beacon, let's see how the narrator
responds when I directly challenge him on it."* (Live experiment in
progress — challenge response to be logged as a follow-up.)

**Observed:** PC hails Vance to confirm he's OK. Vance's voiced reply —
*"I'm fine, just get over here before someone else picks the bones clean"*
— is flatly inconsistent with the inciting incident: a distress beacon,
his voice *"ragged with pain"*, **failing life support**, and him hiding
from something. The narrator wrote him as a brusque salvager, not a dying
man who called for help.

**Root cause (source-verified): the inciting-incident fiction is invisible
to every subsequent narrator call.**
- `postIncitingIncidentCard` (`src/session/incitingIncident.js:119`) posts
  the card with **only** `{ incitingIncidentCard: true }` flags.
- The recent-narration ring (`getRecentNarrationContext`,
  `src/narration/narrator.js:419`) selects messages by
  `flags.narratorCard` + `flags.sessionId` and reads
  `flags.narrationText` — **none of which the inciting card carries**. So
  paced narration, `@scene`, and oracle follow-ups (its three consumers)
  never see the opening fiction.
- The other memory surfaces are equally empty at this point: Vance is an
  **unconfirmed draft** (no entity record / generative tier — F3), the
  World Journal has nothing, and scene truths hold only whatever sidecar
  the incident's own generation happened to emit (the beacon /
  life-support facts were in the prose, not guaranteed into the sidecar).

The narrator voicing Vance genuinely did not know about the beacon.
"I'm fine" is the *expected* output of this plumbing, not stochastic drift
— which also reframes part of **F6**: some of the adversarial coloring may
be the narrator improvising Vance with no backstory available.

**Blast radius — session recap too:** `postSessionRecap` builds from the
same `narratorCard` flags, so the inciting incident is also absent from
"Previously On" recaps. The campaign's opening event is the one beat a
recap must never lose.

**Fix directions for triage (not actioned):**
- **(a) Cheap, high-yield:** flag the inciting card as a narrator card —
  add `narratorCard: true`, `sessionId`, `narrationText` alongside
  `incitingIncidentCard` at post time. One-line-shaped change; puts the
  opening fiction into the recent-narration ring **and** session recaps.
- **(b)** F3's instantiation gives the backstory a durable home (entity
  record + generative tier) that survives beyond the 3-card ring.
- **(c)** Belt-and-braces: have the inciting narrator mode explicitly emit
  sidecar truths for its load-bearing facts (beacon sent, life support
  failing, estranged-mentor history) so they enter scene truths
  deterministically rather than at the model's discretion.
