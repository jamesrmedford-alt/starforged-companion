## F2 — Suggested vow should be a clickable button that creates the vow on the character · 🟡 enhancement

**Reporter:** *"the recommended vow should be a clickable button that will
create that vow in your character card."*

**Desired:** instead of (or alongside) the plain *Suggested vow:* text line on
the inciting-incident card, a button that creates that vow — statement as the
name, suggested rank — as a progress item on the player character, visible on
the character sheet's Progress tab.

**Feasibility (anchored, not yet actioned):** mostly wiring; the hard parts exist.

- The vow is already parsed structurally: `splitSuggestedVow()`
  (`src/session/incitingIncident.js:62`) yields `{ statement, rank }`.
- Creation helper already exists: `createCharacterVowItem(actor, { name, rank,
  vowId })` (`src/character/actorBridge.js:455`) — creates the
  `type: "progress"` / `subtype: "vow"` Item with rank validation (defaults
  `dangerous`) and flag-keyed idempotency.
- Button wiring must use `onChatMessageRender` (`src/system/chatHooks.js`) —
  the V13-002 dual-hook helper, never a raw `renderChatMessage` hook.
- Resolve the target PC via `getPlayerActors()` (FOLDER-002: excludes NPC
  cards, which are also `character`-type Actors).
- Store the parsed `{ statement, rank }` in the chat-message flags at create
  time so the click handler reads flags rather than re-parsing card HTML; the
  message id can serve as `vowId` for free idempotency (re-clicks no-op).
- Plus the usual: unit tests for card render + flags, Quench coverage,
  `packs/help.json` + `CHANGELOG.md` updates.

**Estimate:** small-to-medium.

## F3 — Inciting incident introduces the vow-target NPC but never instantiates them · 🔴 open

**Reporter:** *"the action is good but the followthrough is needed — the
inciting incident introduces an NPC to make a vow to, but doesn't instantiate
them in the entities ui or the actors folder or the world journal."*

**Observed:** the inciting incident invented **Vance** (estranged mentor, the
suggested vow's target) — but after the card posted:
- Entities panel: Connections shows only the pre-existing **Amelia Gray**.
- Actors sidebar: `Sectors / Igneous Spine / NPCs` contains only Amelia Gray.
- World Journal: *"No entries awaiting review"* — no pending lore, no draft.

Vance exists only in the card prose. Notably the narrator **did** ground in
established entities (real ship *Kobayashi 7*, established settlement *Lyra*)
— inventing the mentor was appropriate; the gap is purely follow-through.

**Anchor (verified in source):** `runIncitingIncident`
(`src/session/incitingIncident.js:134`) posts the card directly with **no
detection pass**. `runCombinedDetectionPass` (`src/entities/entityExtractor.js:132`)
is wired into only two paths: the move pipeline (`runPostNarrationPasses` →
`runDiscoveryDetection`, `src/narration/narrator.js:346`) and paced narration
(`runPacedDetection`, `narrator.js:1165`). The `inciting_incident` narrator
mode — like the session-vignette mode — never runs it, so nothing the opening
fiction introduces can reach the draft-Confirm flow.

**Options for triage (not decided):**
- **(a)** Run the standard combined detection pass on the inciting prose —
  Vance arrives as a pending draft; GM Confirms → NPC card (sector NPCs
  folder), Entities panel row, description persisted. Matches the module's
  GM-confirms-entities philosophy.
- **(b)** Treat the vow-target NPC as first-class: the inciting incident
  *knows* it proposed a vow to a person — auto-create (or one-click-create)
  the connection NPC card, the way `make_a_connection` / the sector wizard do.
  Could fold into **F2**'s button: swearing the suggested vow also
  instantiates its target NPC.

## F4 — Time-dependent suggested vows should carry a clock · 🟡 enhancement

**Reporter** (after manually creating the suggested vow on the PC): *"this
incident, autopopulated as a 'before his life support fails', feels like it
might benefit from also having a clock. Need at least a quick pass to see if
there is a time dependence there."*

**Observed:** the suggested vow statement carries explicit time pressure
(*"…before his life support fails"*), and the foundry-ironsworn vow sheet
natively supports exactly this — a **Clock** checkbox right on the progress
item (visible in the reporter's screenshot).

**Anchor (verified in vendor source at the pinned commit):**
- Progress items carry `system.hasClock` / `system.clockTicks` /
  `system.clockMax`; the sheet offers 4/6/8/10/12 segments
  (`vendor/foundry-ironsworn/src/module/vue/progress-sheet.vue:86–119`).
  A vow created by F2's button could arrive with its clock pre-attached —
  `createCharacterVowItem` would just need to accept the extra fields.
- Module-side alternative/complement: tension clocks
  (`src/clocks/clocks.js`, `createClock`) advance against Ask-the-Oracle
  odds — mechanically a good model for "life support failing" — but they
  live in the Clocks panel, not on the character sheet.

**Detection options for "is there time dependence" (not decided):**
- **(a) lead:** the narrator emits it structurally — extend the
  `inciting_incident` output contract
  (`src/narration/narratorPrompt.js:676`, currently `Suggested vow:
  <statement> (<rank>)`) with an optional `Suggested clock: <label>
  (<segments> segments)` line, emitted only when the incident has explicit
  time pressure. The same generation that wrote the deadline already knows
  it's there — zero extra API calls; parse alongside `splitSuggestedVow`.
- **(b)** heuristic regex on the vow statement ("before / until / within…")
  — cheap but brittle.
- **(c)** a Haiku classification call — overkill for one line given (a).

**Relation:** F2 / F3 / F4 are converging into one **inciting-incident
follow-through cluster**: the suggested-vow button (F2) creates the vow,
pre-attaches a clock when the vow is time-dependent (F4), and instantiates
the vow-target NPC (F3).
