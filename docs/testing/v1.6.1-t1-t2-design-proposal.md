# T1 / T2 — Design Proposal (v1.6.1 playtest themes)

**Status:** 📋 PROPOSAL — awaiting decisions before implementation.

This covers the two architectural/subjective themes pulled out of the v1.6.1
playtest findings (`docs/testing/v1.6.1-playtest-findings.md`). The clear bugs
were fixed in PR #145; these two need product decisions first because they trade
off automation against control and touch generation timing + LLM prompts.

---

## T1 — "Entity created & assigned" lifecycle gate

### The problem (findings F3, F6, F7, F8, F9b)

The module **generates eagerly at creation time** — names, flavor prose, art,
journal/Notes stubs, and folder placement all fire before an entity is fully
named, assigned, and carrying its real details. Symptoms observed:

- **F6** — ship Notes referenced the placeholder name "Ship" because flavor was
  generated before the ship was named "Kobayashi IV".
- **F7** — ship flavor invented a carrier/fighters/oversized-engines loadout that
  contradicts the actual Command Vehicle + Research Lab assets (the oracle "roll
  twice" seeds drove prose that the real asset list then contradicts). **F10
  confirmed this propagates** — the narrator reads the ship Notes, so the bad
  flavor becomes working truth in later narration.
- **F8** — settlement Actors land in `Sectors / Unsorted` because they're created
  before the sector is resolvable in `campaignState`
  (`getOrCreateSectorActorFolder` falls back to Unsorted).
- **F3** — settlement Actor sheets are created empty.
- **F9b** — the begin/galley vignette can't use the real ship name.

Root cause: there is no explicit "this entity is now finalized" moment. Compare
the **Session Begin gate** (shipped v1.6.0), which already solved the analogous
problem for narration — nothing fires until the GM presses Begin.

### Proposed direction

Introduce an explicit **finalize step** for entities, after which (and only
after which) generation runs, grounded in the entity's actual sheet data:

```
create (named, typed)  →  assign (sector / owner / assets known)  →  FINALIZE
                                                                      └─► generate flavor / art / stubs
                                                                          (reads the real Actor: name,
                                                                           assets, sector, type)
```

Concretely:

1. **Defer generation** out of the creation path. `seedStarshipActor`,
   settlement creation, and the sector stub pass should not call the narrator /
   art pipeline inline. Instead they create the record and mark it
   `pendingFinalize`.
2. **A finalize trigger.** Options below (decision needed).
3. **Ground generation in the Actor.** The flavor prompt reads the finalized
   Actor's name, **equipped assets** (so F7 can't invent a loadout), sector, and
   type — not just the oracle seeds.
4. **Folder placement at finalize.** Resolve `Sectors / <Sector>` once the sector
   is persisted, so nothing lands in Unsorted (F8).

### Decisions needed for T1

- **D1 — finalize trigger.** Which of:
  - (a) **Explicit GM action** — a "Finalize" button on the entity panel / a
    `!finalize` affordance. Most control, an extra click per entity.
  - (b) **Auto-finalize on a signal** — e.g. when the Actor is renamed off its
    placeholder *and* its assets are set, debounced. Less friction, more magic.
  - (c) **Finalize at sector-wizard completion** for sector-born entities, and
    explicit for ad-hoc ones. Hybrid.
- **D2 — re-generation on change.** If assets/name change after finalize, do we
  regenerate flavor (and burn another art slot), or leave the finalized record
  alone? (Art is already generate-once/locked — see `decisions.md`.)
- **D3 — scope for v1.next.** Ship-only first (highest-value, clearest), or
  ship + settlements + sector stubs together? Ship-only is a smaller, safer slice
  and proves the pattern.

### Relationship to existing scope

This overlaps the **planned** Entity → Actor Migration scope
(`docs/entities/entity-actor-migration-scope.md`), which already makes Actors the
source of truth and adds the folder tree. T1's "ground generation in the Actor"
is the natural complement. **Recommendation:** fold T1 into that scope's first
implemented slice rather than building a parallel mechanism.

## T2 — World Journal / Chronicle auto-capture salience

### The problem (findings F15, F17, F20, F21; F18 is the routing half)

Auto-capture is **over-eager** and records transient scene beats as permanent
entries:

- **F17 / F21** — World Journal *Lore* fills with moment-to-moment minutiae
  ("Container C-47 Scorch Marks", "External Airlock Cycling Sequence Initiated").
- **F20** — the Chronicle captures the same beats, all tagged "Revelation". The
  writer (`src/character/chronicleWriter.js`) fires on **every** paced/move
  narration and always emits a one-sentence entry — there is no salience gate.
- **F15** — *Threats* over-captures scene complications as world threats.
- **F18** (routing, related) — the *Session Log* was blank (now fixed: it's
  written on End Session) while the scene beats that belong there were going to
  Lore as separate pages.

The reporter's framing: Lore "looks like things that could have been a running
session log, where each new page is really just an appendment to the existing
scene."

### Proposed direction

Three independent levers (can ship separately):

1. **A salience threshold.** Give the detector/chronicle an explicit bar for
   "worth a permanent record" vs "scene texture". Implementation options:
   - prompt-level: instruct the model to emit `salience: high|scene` and only
     persist `high`;
   - confidence reuse: the entity detector already returns
     `confidence: high|medium|low` and drops `low` — extend the same idea to WJ
     lore/threats with a configurable floor.
2. **Scene-transient vs campaign-durable routing.** Route low-salience beats into
   the **running Session Log** (append to one page per session) instead of
   spawning a Lore page each. Reserve Lore for durable world facts. This is the
   F18 insight turned into a routing rule.
3. **Chronicle salience gate.** Stop writing a chronicle entry on *every*
   narration; only when the beat clears the bar. (Today: unconditional per the
   `chronicleAutoEntry` toggle.)

### Decisions needed for T2

- **D4 — control surface.** A single "auto-capture sensitivity" dial
  (Quiet / Balanced / Everything), or per-channel toggles (Lore / Threats /
  Chronicle each with their own threshold)? A dial is simpler; per-channel is
  more precise and matches the existing per-feature settings style.
- **D5 — default.** Should defaults lean **conservative** (record less; the GM
  promotes scene beats up) given the playtest reaction, or stay permissive? The
  WJ panel already has a Confirm step for Lore, so under-capturing is cheap to
  correct, over-capturing creates cleanup.
- **D6 — retroactive.** Do nothing to existing over-captured entries, or offer a
  one-time "sweep scene minutiae into the session log" cleanup? (Probably do
  nothing; out of scope for the first slice.)
- **D7 — running Session Log shape.** One page per session that the detector
  **appends** to (vs. the current End-Session summary page). Append-during-play
  is a bigger change than the End-Session write just shipped for F18.

### Recommended first slice (lowest risk, highest signal)

- Add a **salience field** to the WJ-lore + chronicle prompts and a single
  world setting `autoCaptureSensitivity` (default **Balanced**, leaning
  conservative). Persist only entries at/above the threshold.
- Keep Threats and the Session-Log-append routing as a **follow-up** once the
  threshold behavior is validated in play.

---

## Suggested sequencing

1. **T2 salience first** — it's prompt + setting + a persist gate, no storage
   migration, and directly addresses the most-repeated playtest complaint.
2. **T1 ship-only finalize** — proves the "generate after finalize, grounded in
   the Actor" pattern on the highest-value entity, ideally as the opening slice
   of the Entity → Actor Migration scope.
3. **F16 live-verification** (from PR #145) — confirm the dedicated control group
   renders before relying on it.

All three want a product decision (the **D1–D7** above) before code. Once those
are answered I can implement each as its own gated PR with regression tests, same
as the PR #145 bug fixes.
