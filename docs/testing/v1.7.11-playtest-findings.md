# v1.7.11 Playtest Findings

Seven findings from the v1.7.11 playtest (2026-06-13, fresh world via ✦ Playtest
Quickstart). All seven were diagnosed to root cause and fixed together on
`claude/admiring-carson-qlzr7h` (PR for v1.7.12).

Status key: 🟢 fixed/resolved · ✅ positive.

---

## A — Quickstart PC opens with the classic Ironsworn sheet · 🟢 fixed

**Reporter:** *"the PC card is no longer the starforged one, it has changed to
the ironsworn one."* (Dex Sedano showing Bonds / Conditions / Banes / Burdens.)

**Root cause:** same class as the v1.7.10 NPC-sheet finding, new call site.
foundry-ironsworn defaults every `character` actor to the classic sheet; the
system's create-dialog pins the Starforged one, but ✦ Playtest Quickstart
(PR #168) creates the PC directly (`createQuickstartPc`) and never pinned
`core.sheetClass`. The v1.7.10 backfill only touched NPC cards, so Dex wasn't
repaired either.

**Fix:** quickstart pins the Starforged sheet at PC creation; the ready-time
repair (`backfillNpcCardSheets`) now covers all `character` actors lacking an
explicit sheet choice (PCs included), so existing quickstart PCs self-heal.
Deliberately re-sheeted actors are left alone.

## B — NPC Characteristics show raw HTML markup · 🟢 fixed

**Reporter:** *"the NPC has the correct card, but the characteristics have
html-like markers in it."* (`<p><em>Oracle-seeded…</em></p><ul><li>…` shown
literally.)

**Root cause:** the seed wrote `buildConnectionBiographyHtml(...)` (an HTML
`<ul>`) into `system.biography`. The Starforged sheet renders Characteristics
in a **plain `<textarea>`** (`sf-characterheader.vue`), so the HTML displayed
verbatim. Latent since FOLDER-002 seeding — invisible before A because the
classic sheet's Characteristics field is a rich-text editor.

**Fix:** Characteristics is now plain text (`buildConnectionCharacteristics`).
The Notes tab (a rich-text field) keeps its HTML via a separate builder.

## C — A ship token placed on the map is invisible to positioning · 🟢 fixed

**Reporter:** *"I have my ship token on the board, but the narrator doesn't see
it that way."*

**Root cause:** all command-vehicle token logic gated on
`flags[MODULE].commandVehicle`, stamped only by the module's own auto-placement.
Dragging the ship actor from the sidebar makes a plain token with no flag, so
the v1.7.10 placement/reposition hooks, drag-to-Set-a-Course, and the
fiction→token sync all ignored it. Compounded by quickstart building the sector
(phase 2) before the ship existed (phase 4), so nothing was ever auto-placed.

**Fix:** `isCommandVehicleToken` recognises a token by flag **or** by its
`actorId` resolving to the registered command vehicle; placement stamps the
flag so the sync can find it later. Quickstart places the ship token after the
ship phase (4b), and the createToken hook records the starting position.

**Positive:** the v1.7.11 "not yet established" guard worked exactly as
designed in the interim — `@scene where am I` refused to invent coordinates and
handed the anchor back to the player.

## D — Sector map camera gets stuck zoomed in · 🟢 fixed

**Reporter:** *"I had my character icon highlighted. when I spoke, it zoomed in
on it, and now I can't reset the scene … I was unable to pan or zoom out … I can
zoom in further, and then back out to where I started, but not back to the
original size and no panning."*

**Root cause (corrected after an initial wrong guess):** not pan-to-speaker and
not a frozen canvas — the camera was **clamped**. `sceneBuilder` set the scene's
`padding: 0`, and padding is the buffer the canvas pans/zooms into. Zero padding
pins the camera to the image edge: you can zoom in, but never pan or pull back
to a full-map framing. (The reference doc's own scene example used `0.1`; `0`
was a regression.) The scene also captured no initial view.

**Fix:** `padding: 0.1` (the black buffer matches the starfield) restores a
usable camera; a captured `initial` view (centred, fit-scale ≤ 1) gives a
reliable "whole map" home. Per the reporter's choice (option b), the pan itself
is kept.

## E — NPCs have no established gender; art/narrator/audio each guess · 🟢 fixed

**Reporter:** *"we should also include pronouns on NPCs. The NPC generated as
distinctly male leaning, but the narrator has them canonically female. Picking a
gender intentionally should help clarify."*

**Root cause:** the connection schema had a `pronouns` field but nothing
populated it. The portrait prompt leaned on role/look (male-presenting art); the
narrator invented pronouns per call ("her"); they diverged.

**Fix:** the seed rolls a pronoun set once, stores it on the record **and**
`system.pronouns`, leads the portrait prompt with a matching descriptor
("a woman/man/person"), threads it into the seeded Notes prose, and surfaces it
on the narrator entity card so live narration stays consistent.

## F — Audio uses one voice for every NPC, regardless of gender · 🟢 fixed

**Reporter:** *"the audio went and chose a distinctly male voice. I don't think
this is narrative choice, this just sounds like failure to identify and
communicate."*

**Root cause:** the audio layer used a single `audio.npcVoiceId` for all
`<npc>` speech — no per-NPC resolution, no gender input. Exactly "failure to
identify": it was never told who was speaking.

**Fix (builds on E):** optional pronoun-keyed NPC voices
(`audio.npcVoice{Feminine,Masculine,Neutral}`); a card's focal connection
(`matchedEntityIds`) selects the matching voice, falling back to the default
NPC voice when unset, absent, or when matched NPCs disagree. Strictly an
improvement over one-voice-for-all; multi-NPC-per-card stays a documented
limitation (audio is per-card, not per-segment).

## G — Asset "improve the result" (Fugitive) couldn't be applied post-roll · 🟢 fixed (new feature)

**Reporter:** *"My character has the Fugitive trait … Need to wire in that kind
of transaction as a possibility when you use the trait (it was highlighted when
I chose the roll, which was good, but I couldn't adjust the result)."*

**Root cause:** the ability scanner is **pre-roll only** — it surfaces
applicable abilities as confirm-dialog checkboxes and folds their `+N` into the
roll. Fugitive's "improve the result to a strong hit (then fill a clock)" is a
**post-roll** transaction, and the pipeline had no post-roll step except Burn
Momentum.

**Fix:** a post-roll affordance modelled on Burn Momentum
(`src/moves/improveResult.js`). The scanner now classifies result-improvement
abilities (`extractResultImprovement`) and carries the ability's own clock
(`hasClock`/`clockTicks`/`clockMax`). When a weak hit or miss has an applicable
ability, the move-result card shows **✦ Improve to Strong Hit (Fugitive —
fill 1 of 4)**; clicking upgrades the outcome, reverses/re-applies meter
consequences, advances the asset ability's clock as the cost, supersedes the
stale narration, and re-narrates. Bounded to non-progress moves, matching burn.

## Notes

- Findings A/B/E/F are one thread: an NPC/PC's *identity* (sheet, gender) must
  be established once and propagate to every surface (sheet field, art prompt,
  narrator context, audio). Each surface guessing independently was the failure.
- D is the cautionary entry on diagnosis discipline: the first hypothesis
  (pan-to-speaker) was a pattern-match that didn't fit "can't pan or zoom out";
  the reporter's follow-up ("zoom back only to where I started") pointed at a
  camera-bounds clamp, which traced to `padding: 0` in our own code.
